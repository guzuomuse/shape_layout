### 二、 媲美/超越 Figma 与 InDesign：Phase 3 必修的三个杀手级功能

为了让 `shape_layout` 彻底跨过“好用的工具”，成为“行业级别的柔性制造排版编译器标准”，我建议我们在 Phase 3 引入以下三个极具商业张力和技术壁垒的功能：

#### 1. 均匀分布/自动间距（HAlign::SpaceBetween / Auto Gap）
* **用户的生财痛点**：
  在设计高端不干胶贴纸或吊牌时，设计师不希望间距（gap）是死板的 5mm。他们希望：**“Logo 贴着最左边，二维码贴着最右边，中间的 3 个元素能自动平分剩下的空间，形成完美的黄金分割两端对齐。”**
* **如何利用你现存的 `kasuari` 底座极简实现**：
  在 `HAlign` 中引入 `HAlign::SpaceBetween` 变体。我们在 `solve_row_x` 注入间距约束时，不需要传入硬编码的 `config.gap`，而是通过**一个统一的弹性变量**来约束：
  ```rust
  HAlign::SpaceBetween => {
      if n > 1 {
          // 1. 注册一个统一的弹性间距变量
          let gap_var = Variable::new();
          // 限制最小间距（REQUIRED）
          solver.add_constraints([gap_var | GE(Strength::REQUIRED) | config.gap]).unwrap();
          
          // 2. 将所有相邻元素的间距，死死锁在这个弹性变量上（REQUIRED）
          // 所有的间距被迫在数学上绝对相等
          for i in 0..(n - 1) {
              solver.add_constraints([(left_vars[i + 1] - right_vars[i]) | EQ(Strength::REQUIRED) | gap_var]).unwrap();
          }
          
          // 3. 强行拉满两端
          solver.add_constraints([left_vars[0] | EQ(Strength::REQUIRED) | interval_l]).unwrap();
          solver.add_constraints([right_vars[n - 1] | EQ(Strength::REQUIRED) | interval_r]).unwrap();
      }
  }
  ```
  *不需要任何除以零、浮点数越界碰撞代码，`kasuari` 会在微秒内，以数学上的绝对精准度，算出那个最完美的、等分排满区间的间距 `gap_var` 数值。这就是约束求解器的终极魅力。*

#### 2. 真正的“异形内缩边距”（SDF / Contour-based Padding）
* **用户的生财痛点**：
  当前我们的 `LayoutConfig` 虽然有 `padding_left` / `padding_right`，但它们只是在 1D 切片出来的 `[L, R]` 区间两端加上固定的位移。
  如果容器是一个非常复杂的异形（比如心形），**单纯的左边距和右边距，无法阻止文字在转角处贴近心形倾斜的边缘**，视觉上非常局促、不安全。
* **如何优雅解决（降维打击）**：
  我们**不需要**去写复杂的 2D 干涉计算！
  * 别忘了，你有了 **`cavalier_contours`** 这个等距偏移神器。
  * 当用户设置了 `padding = 3.0mm` 时。
  * 你的引擎在初始化 `RangeGenerator` 之前，**直接调用 `cavalier_contours` 对容器的 `BezPath` 整体向内收缩 3mm 得到一个“内宿多边形（Inner Safe Polyline）”**。
  * 然后，将你的扫描线 `RangeGenerator` 跑在这个“内宿多边形”上。
  * **结果**：算出来的所有 `[L, R]` 区间，**天然、绝对、100% 沿着心形起伏的不规则边缘保持着完美的 3mm 呼吸感距离**。
  * 此时，用户设置 `padding_left / padding_right = 0`，排版出来的画面干净得像教科书。这在工业界叫“轮廓出血保护”。

#### 3. 垂直排版组（Nestable VGroup / Figma-style Columns）
* **用户的生财痛点**：
  我们目前的布局是全局 V-Flow（行向下走），行内 H-Flow（元素向右走）。
  如果用户希望：**在行李牌的左边并排一个圆形 Logo，右边垂直叠放两行文字（第一行是名字，第二行是电话）**。
* **如何优雅实现**：
  我们在 `LayoutElement` 的类型中，允许它本身是一个 **`VGroup`（垂直组）**。
  * `VGroup` 内部的子元素是垂直堆叠的。
  * 在前置分行计算中：`VGroup` 的首选高度 $H = \sum(\text{子元素高度}) + \text{间距}$，首选宽度 $W = \max(\text{子元素宽度})$。
  * 我们的排料引擎依然把它当成一个普通的“大矩形”去进行 `layout_rows()` 拼版。
  * 当这个大矩形解出 $(X, Y)$ 坐标后，大矩形内部的子元素在后台执行一个最简单的、无视异形的垂直累加定位。
  * **价值**：支持了垂直组（VGroup），你就完全拥有了在不规则容器内，构建任何复杂图文/条码混排版面的全部自由度。

## 🔬 我的深度分析：这是「设计空白」还是「Bug」？

### 角度 A：kasuari 的实际行为

kasuari 是一个**确定性**的 Simplex 求解器。对于同一个约束系统，给定相同的初始条件，它总是产生**相同的结果**。这个结果是：

1. **满足所有约束**（REQUIRED/STRONG/MEDIUM/WEAK 按优先级）
2. **最小化总惩罚值**（加权最小二乘）

在 `[Fill_A, Fill_B]` 场景中，如果没有 WEAK 宽度偏好：
- 两个 Fill 都没有"偏好宽度"的约束
- kasuari 会把 `right_0` 放在**使惩罚值最小**的位置
- 由于两边对称，结果很可能是**中间值**（接近 47.5）

**结论**：结果不是"随机的"，而是**确定性的**，只是这个确定性结果不一定符合用户的"直觉期望"（均分）。

### 角度 B：与 CSS Flexbox 的对比

CSS Flexbox 中，`flex-grow: 1` 的多个元素会**按比例分配**剩余空间。这个比例是基于 `flex-grow` 值的。

当前 `shape_layout` 的 `SizeStrategy::Fill` 相当于 `flex-grow: 1`（无权重差异），所以用户**直觉上期望**两个 Fill 元素均分剩余空间。

### 角度 C：不加 WEAK 的风险

如果 kasuari 的内部实现（Simplex 初始化）把 `right_0` 放在边界（如 10 或 85），那么：
- 一个 Fill 被压到 `min_width`
- 另一个 Fill 吃掉所有剩余空间

这在**数学上正确**，但在**语义上不可接受**。

**关键问题**：kasuari 的初始化策略是什么？

kasuari 的 `Solver::new()` 创建变量时，初始值为 0。在 Simplex 求解中，如果没有偏好约束，求解器倾向于**把变量推向使目标函数最小的方向**。对于对称约束，这通常意味着**中间值**。

但为了**绝对确定性**，我们不能依赖 kasuari 的内部实现细节。
