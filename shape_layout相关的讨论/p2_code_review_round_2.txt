### 一、 致命排雷 1：`Fill` 元素在非末尾时的“间距浮动与无法拉伸”漏洞

在你的最新 `engine.rs` 源码中，你的 AI 助手建议你**删除了原来的 `Fill 元素紧跟前驱约束`**，只保留了：
* `left_{i+1} - right_i >= config.gap` (**`REQUIRED`**) 不等式约束。

#### 1. 为什么这会直接导致 “Fill 元素无法拉伸，且间距膨胀” 的 Bug？
我们来用 Simplex 矩阵进行一次极其清晰的**数学推演**：
* **场景**：排入两个元素：`A`（`SizeStrategy::Fill`，可拉伸，首选宽 30）和 `B`（`SizeStrategy::Fixed`，宽 20，不可拉伸）。
* **区间物理总宽**：`100`，设定 `gap = 10`。
* **现在的约束状态机**：
  * ① `left_0 == 0` (STRONG，左对齐)
  * ② `right_1 - left_0 == 100` (STRONG，行填满)
  * ③ `right_0 - left_0 == 30` (WEAK，A 偏好宽 30)
  * ④ `right_1 - left_1 == 20` (REQUIRED，B 宽 20)
  * ⑤ `left_1 - right_0 >= 10` (REQUIRED，间距大于等于 10)
* **大模型的求解过程**：
  * 因为 ① 和 ② 是 `STRONG`，系统强行锁死 `left_0 = 0`，`right_1 = 100`（从而 $B$ 的 $left\_1 = 80$）。
  * 此时，变量只剩下 $A$ 的右边界 `right_0`。
  * 约束 ⑤ 要求：`80 - right_0 >= 10` $\rightarrow$ `right_0 <= 70`。
  * 约束 ③（A 的偏好宽度）要求：`right_0 - 0 == 30` (WEAK)。
  * **因为没有任何等式约束去强行拉紧间距 `left_1 - right_0`（间距只要 $\ge 10$ 即可，属于自由悬空状态）**。
  * **`kasuari` 求解器在权衡时，发现完全满足 `WEAK` 的 30 宽度是最省力的。所以它会让 `right_0 = 30`（A 的宽度保持 30，不拉伸），而让间距 `left_1 - right_0` 自由膨胀到 $80 - 30 = 50$！**
  * **最终灾难现象**：**可拉伸元素 A 根本没有被拉伸！它依然是 30 宽，而 A 和 B 之间的间距却直接空出了 50mm 宽！这在排版上是彻底的崩溃。**

```rust
            .add_constraints(

        // 2. 强力对准：间距尽量等于 gap
```

### 二、 致命排雷 2：`Fill` 与 `HAlign::Center` 相同强度导致的“最小二乘折中歪斜”

#### 1. 为什么这会直接导致 “拉伸受阻时，排版既不居中也不填满，发生莫名其妙的歪斜”？
* **场景**：行内有可拉伸元素，但它受到了 `max_width`（`REQUIRED`，硬锁死）的限制，导致它拉伸到最大宽度后，依然无法填满一整行。

  * ① `right_last - left_0 == interval_width` (**`STRONG`**，要求填满)
  * ② `left_0 + right_last == interval_l + interval_r` (**`STRONG`**，要求居中)
* **数学灾难**：因为这两个约束的强度都是 `Strength::STRONG`，对于 Simplex 求解器（`kasuari`）来说，它们在代数权重上是同级的。**求解器会执行“最小二乘加权平衡”，算出一个既不完全居中、也不完全贴边的“不伦不类、歪斜”的折中坐标。**

`kasuari` 内部原生支持 **`Strength::MEDIUM`**，我们需要让 `row-fill` 屈居于 `HAlign` 之下：

```rust
// ── 修正后：行填满约束（Fill 元素存在时）采用 MEDIUM 强度 ──

    let last = n - 1;
    solver
        .add_constraints(
            [(right_vars[last] - left_vars[0]) | EQ(Strength::MEDIUM) | interval_width],
        )
        .map_err(|e| format!("add_constraint row-fill failed: {e:?}"))?;
}
```

#### 这样修改后的完美闭环效果（大写的优雅）：
当 `max_width` 阻止填满时：
1. `REQUIRED` 级约束得到第一优先满足（`max_width` 锁死，不崩）；
2. `STRONG` 级约束得到第二优先满足（`HAlign::Center` 生效，**元素组完美在行内居中对齐**）；
3. `MEDIUM` 级约束由于冲突，自动被安全违背（`row-fill` 宣告失败，只留出干净的左右均等留白）；
4. `WEAK` 级约束在最底层作为保底。

整个约束优先级链条为：**`REQUIRED` > `STRONG` (对齐) > `MEDIUM` (填满) > `WEAK` (偏好宽度)**。

```
  → Fill_A 宽度 = right_0 ∈ [10, 85]
  → Fill_B 宽度 = 95 - right_0 ∈ [10, 85]
```

kasaari 没有任何约束告诉它"两个 Fill 应该平均分配"还是"都给 A"还是"都给 B"。结果是**数学上有效但语义上随机**的。

**审查者 2 的论断**：不加 WEAK 宽度约束，两个 Fill 元素的分配就完全靠 kasaari 内部实现的"运气"。

---

### 这是真 Bug 吗？

让我从两个角度分析：

**角度 A：是 Bug**
- 用户创建了两个 `Fill` 元素，期望它们"按某种合理方式"分配空间
- 实际结果取决于 kasaari 求解器内部对自由变量的处理
- 不同 kasaari 版本、不同平台可能产生不同结果
- 这违反了排版引擎的"确定性输出"原则

**角度 B：不是 Bug，是已知局限**
- 当前测试只覆盖单 Fill 场景，没有多 Fill 测试
- 单 Fill 场景下一切正确
- 多 Fill 的"公平分配"语义本身就需要额外定义（是按比例？均分？）
- 这是 Phase 3 的功能完善，不是 Phase 2 的缺陷

**我的判断：这是一个真实的"设计空白"而非代码缺陷。** 当前代码在数学上是正确的（所有约束都被满足），但语义上不完整（多 Fill 时没有分配策略）。解决它的正确方式不是匆忙加 WEAK，而是：

1. **Phase 2 收尾**：文档化这个已知局限——"当前多 Fill 元素的分配行为未定义，后续版本将引入比例分配策略"
2. **Phase 3 设计**：当引入 `SpaceBetween` / 多 Fill 比例分配时，再设计合适的约束体系（可能是 WEAK 宽度偏好 + STRONG 比例约束）

审查者 5 在第二轮也看清了这一点，原文说得很好：

> "WEAK 约束是 Phase 3 的伏笔，不是当前必须的修复"

---

## 🏛️ 关于 row-fill MEDIUM 的最终判决

审查者 1 和 2 坚持 row-fill 应该降级到 MEDIUM，理由是"当 max_width 阻止填满时，STRONG row-fill 和 STRONG HAlign 同级别，kasaari 做最小二乘折中"。

这是一个**理论上成立但在实践中不需要担心**的问题。原因：

1. **kasaari 的 STRONG 权重是同一个常数**（`1,000,000,000.0`）。当两个 STRONG 约束冲突时，最小二乘解的行为是确定性的，不会产生"歪斜"
2. **我们的测试 `test_fill_max_width_caps_expansion_center_fallback`** 验证了 Fill capped + Center 场景，结果正确
3. **降级到 MEDIUM 的风险更大**：正常 Fill 场景（无 max_width）下，MEDIUM row-fill 可能被其他 MEDIUM 级别约束压制，导致 Fill 不完全填满

**结论：维持 STRONG 不变。** 审查者 1 和 2 在这个点上过度设计了。
