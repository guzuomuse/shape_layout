### 规则一：外壳是“绝对硬边界（Hard Ceiling）”，不可自底向上逆向膨胀

在网页设计（如 HTML/CSS/Figma）中，一个按钮变长了，可以把整个网页往下推几千像素，这叫“自底向上的流式膨胀”。

但在 **B2B 物理制造（VDM）** 中，这是绝对禁止的。
* **物理硬伤**：工件的物理外形（那个心形、不锈钢牌、洗发水瓶身）是**已经锁死、甚至已经切好的物理实体**。
* **Figma 和 InDesign 的做法**：
  * **容器硬锁定（Fixed Parent Container）**：最外层的 Container 必须是 `Fixed` 尺寸。
  * 当内部文字变长时，文字可以自适应，但它**绝对不能把最外层的容器撑大**。
  * **结果**：当 *Alexander* 太长，且已经缩到了最小字号（`min_width`）依然放不下时，排版编译器**宁愿报错并把该元素丢进 `unplaced` 列表，也绝对不会强行把标签撑破或把其他元素挤飞。** 这在工业上叫“安全熔断（Fail-Safe）”。

---

### 规则二：禁用“无感动态折行（No-Wrap by Default）”，行内高度强隔离

在网页布局中，如果一行放不下，子元素会自动换行（Wrap）到下一行，从而把下方的所有行像多米诺骨牌一样往下推，导致整个排版瞬间解体。

* **Figma 的做法**：
  * 在 Auto Layout 中，除非设计师手动开启了 `Wrap`（换行）属性，否则默认全部是 **`No-Wrap`（单行流）** [1.1.5, 1.1.7]。
* **我们的 VDM 编译做法**：
  * 我们的 `LayoutRow`（行）在底层设计上，**默认是绝对禁止动态折行的**。
  * 如果行内元素合起来超出了当前的 `[L, R]` 区间，
    * 如果元素是 `shrinkable`，则在行内原地压缩（字号变小）；
    * 如果不可压缩，该元素直接被剔除到 `unplaced` 队列中，下一个元素绝对不会“掉落”到下一行去推开别人。
  * **价值**：这保证了每一行的垂直高度 $H_{row}$ 在分行阶段就已经被**静态锁定**，行与行之间互不干扰，彻底切断了垂直方向的多米诺骨牌连锁反应。

3.  **引入“冲突解决机制”**：自适应规则有时会互相冲突，造成你担心的“尴尬”局面。InDesign的应对策略值得借鉴：它会根据一套内置的优先级逻辑**自动解决冲突**。例如，当子元素设为`填充`，而父容器设为`适应内容`时，可能触发父容器变为`固定`尺寸的自动裁决。**系统会通过输出警告（`LayoutWarning`）来告诉你发生了什么，而不是擅自破坏规则。**

4.  **提供“紧急出口”：绝对定位**：当完全的自适应规则无法满足特定设计需求时，需要提供“忽略自动布局”的选项，允许将某些元素从布局流中取出，进行自由定位。这是**可控性**的最后一道防线。

### 📐 `shape_layout` 的“自适应”该做多深？

基于以上分析，你的`shape_layout`不必、也不应该去解决所有美学问题。它的使命是**稳定、高效地执行规则**。因此，建议将自适应能力分为三个层次：

*   **核心层 (必须做到100%)**：**物理安全的自适应**。即你已经在做的：确保元素在**不越界、不重叠**的前提下，满足用户指定的`HAlign`、`VAlign`、`SizeStrategy`等规则。这是`shape_layout`的底线和数学保证。

*   **规则层 (扩展意图表达)**：**引入更丰富的规则集**，用于表达复杂的自适应。例如，借鉴Figma，可以增加：
    *   **`SizeStrategy::Hug`**：让容器尺寸适应子元素。
    *   **`SizeStrategy::FillRatio(fr)`**：允许元素按比例分配空间。
    *   引入**冲突解决机制**，像InDesign一样，在规则冲突时尝试自动裁决，并将决策（或失败原因）通过`LayoutSolution`的`warnings`反馈给上层。

*   **美学层 (交由用户与AI)**：**“排版是否好看”，是用户的审美特权，也是AI学习的空间。** `shape_layout`只需忠实地执行规则，确保物理和逻辑的正确性。一个好或坏的视觉结果，源于其上层提供的规则质量。`

### 💎 回到你的核心问题：失控的正反馈循环怎么破？

你描述的“一个元素变化引发全局震荡”正是典型的规则冲突或无限递归。解决办法是：

1.  **系统必须有明确的“规则仲裁者”**：当冲突发生时（例如，父容器要收缩，子元素要填充），系统必须根据预设的优先级（如InDesign的自动解析逻辑）做出明确决策，而不是陷入无解循环。这个决策结果应通过`warnings`告知调用者。
2.  **提供“绝对定位”作为终极控制手段**：当用户对自动结果不满意时，允许他将特定元素从布局流中“解放”出来，进行手动微调。这确保了**审美话语权始终掌握在人类手中**。

最终的结论是：**`shape_layout`不是要让排版“自动变美”，而是让排版“在可控的规则下自动适应”。** 它的价值在于，让用户（或AI）能够用清晰的规则语言，驱动一个确定性的、物理安全的布局引擎。

这解决了工业制造中最核心的“一致性”和“可靠性”问题。至于“是否好看”，则是在此基础上，由人类的审美和AI的规则优化共同去解决的问题。你之前的“钳制+重排”方案，其实就是为这个“确定性”设立了最后一道安全阀。

✅ 正确目标：有限自适应

“在用户定义的约束范围内，自动调整元素的大小和位置。”

Figma 的 Auto Layout 是有限自适应的典范。

Figma 的自适应规则：

规则 说明

固定宽高 元素大小不变

Hug 元素大小随内容变化

Figma 的自适应是“可预测的”，不是“神奇的”。

五、那你的 shape_layout 应该怎么做？

✅ 核心原则：自适应 = 约束求解，不是自由变形。

自适应类型 是否支持 说明

宽度自适应（Hug） ✅ 元素宽度随内容变化

高度自适应（Hug） ✅ 元素高度随内容变化

填充自适应（Fill） ✅ 元素填充剩余空间

字号自适应 ❌（Phase 3 可选） 字号随容器变化

关键：自适应的“度”由用户控制，不是由引擎决定。

六、那“多米诺骨牌效应”怎么控制？

你担心的这个问题非常真实：

“一个容器变化了，其他元素像多米诺骨牌一样都跟着变化，最终布局面目全非。”

解决方案：分层约束

层级 约束强度 说明

L1：用户意图 REQUIRED 对齐方式、间距、比例

L2：元素关系 STRONG 父子关系、兄弟关系

L3：自适应 WEAK 宽度、高度、字号

当自适应导致布局崩坏时，优先保留 L1，牺牲 L3。

具体做法：

// 用户意图（REQUIRED）
solver.add_constraint(title.x == container.center_x)?.required();
solver.add_constraint(body.left == title.left)?.required();

// 元素关系（STRONG）
solver.add_constraint(body.top == title.bottom + 10)?.strong();

// 自适应（WEAK）
solver.add_constraint(title.width == title.content_width)?.weak();


当内容过长时：
1. 标题宽度自适应（WEAK）→ 被牺牲
2. 标题保持居中（REQUIRED）→ 保留
3. 正文保持左对齐（REQUIRED）→ 保留
4. 正文在标题下方（STRONG）→ 保留

布局变了，但意图没变。
