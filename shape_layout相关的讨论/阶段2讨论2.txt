### 一、 冲突 1 裁决：采用 Figma 的 `SizeStrategy`，彻底给 HAlign 与 Sizing 减负

在关于 “Stretch 状态下 `HAlign` 会不会冲突” 的博弈中：

* 方案三（【回复 7】提出）：**引入 `SizeStrategy` 尺寸策略枚举。这是世界级的设计。**

#### 为什么 `SizeStrategy` 是终极解法？
在 Figma 中，对齐（Alignment）和尺寸策略（Sizing）是完全正交（解耦）的两个维度。
如果我们引入 **`SizeStrategy`**，可以把之前混乱的 `stretchable`、`shrinkable` 属性彻底收缩为一个干净、无状态的枚举：

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum SizeStrategy {
    /// 固定宽度（对应 Figma 的 Fixed）
    Fixed {
        preferred: f64,
        /// 空间不够时是否允许收缩
        shrinkable: bool,
    },
    /// 撑满父容器/行宽（对应 Figma 的 Fill Container）
    /// 它天然具有“能屈能伸”的属性，由 kasuari 动态拉伸分配
    Fill,
}
```
#### 对 `solve_row_x` 的数学模型净化（结合【回复 2】的四态真值表）：
采用 `SizeStrategy` 后，我们在向 `kasuari` 注入宽度偏好约束时，逻辑变得极为干净和防御性极强：

1. **若为 `SizeStrategy::Fixed { preferred, shrinkable: false }`**：
   * 注入：`right - left == preferred` (**`REQUIRED`**) ──> 绝对死板，宁死不弯。
2. **若为 `SizeStrategy::Fixed { preferred, shrinkable: true }`**：
   * 注入：`right - left <= preferred` (**`REQUIRED`**) 和 `right - left == preferred` (**`STRONG`**) ──> 上限压死，允许往下缩。
3. **若为 `SizeStrategy::Fill`**：
   * 注入：`right - left == preferred` (**`WEAK`**) ──> 随波逐流。
   * 同时，由于行内有 `Fill` 元素，我们对整行施加：`right_last - left_0 == interval_width` (**`STRONG`**) ──> 强行把两端拉满，多余的空间全部自动均摊给所有 `Fill` 元素。
   * 必须无条件叠加：`right - left >= min_width` (**`REQUIRED`**) 保护。

### 三、 ⚠️ 隐藏的惊天巨坑：基线对齐下的“动态行高计算”

你搜集的讨论中，**【回复 7】指出了一个全行业 90% 的排版引擎开发者都会踩进去的终极深坑！**

在普通排版中：行高 $H_{row} = \max(\text{elements.heights})$。
但是在 **`VAlign::Baseline`（基线对齐）** 的行内，这个公式是**完全错误且会导致文字切头、越界**的！

#### 物理干涉分析：
假设这一行有两个元素：
1. **大文本 A**：高 20，`baseline = 16` (距离顶部 16，距离底部 4)。
2. **小文本 B**：高 10，`baseline = 2` (距离顶部 2，距离底部 8，比如只有下沉字母 `g, p, y`)。

我们将它们在基线上对准（对准线为 $Y_{baseline}$）：
* A 的顶部在：$Y_{baseline} - 16$
* B 的底部在：$Y_{baseline} + 8$
* **整行的物理最高点**：由 A 的顶部决定，在 $Y_{baseline} - 16$。
* **整行的物理最低点**：由 B 的底部决定，在 $Y_{baseline} + 8$。
* **整行需要的物理总高度**：$16 + 8 = 24$。

如果我们用传统公式 $H_{row} = \max(20, 10) = 20$。
**后果**：整行由于物理干涉，**底部会多出来 4mm 的空间无法被扫描线感知，直接在下方越界、切头、甚至重叠！**

#### 完美的物理公式（【回复 7】提出，请务必合入 `engine.rs`）：
在 `VAlign::Baseline` 模式下，整行的物理行高计算必须改为：
$$\text{Max\_Baseline\_Offset} = \max_{i} ( \text{element}_{i}.\text{baseline} )$$
$$\text{Max\_Descender\_Offset} = \max_{i} ( \text{element}_{i}.\text{height} - \text{element}_{i}.\text{baseline} )$$
$$H_{row} = \text{Max\_Baseline\_Offset} + \text{Max\_Descender\_Offset}$$

这个公式，是 **Figma 和 InDesign 能够进行高精度排版的“镇山之宝”**。

### 1. 关于“伸缩性”的终极统一：四态真值表 (The Matrix)

大家纠结的本质是：**当空间不够或空间太多时，谁先动？**

不要去写复杂的分配算法，直接给 `kasuari` 喂这四种“性格”：

| 属性组合 | 物理性格 | 给 Solver 的指令 (除 min/max 外) |
| :--- | :--- | :--- |
| **不可缩 + 不可拉** | **钢铁** | `width == preferred` (**REQUIRED**) |
| **可缩 + 不可拉** | **弹簧(向内)** | `width <= preferred` (**REQUIRED**) + `width == preferred` (**STRONG**) |
| **不可缩 + 可拉** | **弹簧(向外)** | `width >= preferred` (**REQUIRED**) + `width == preferred` (**WEAK**) |
| **可缩 + 可拉** | **流体** | `width == preferred` (**WEAK**) |

**为什么要这样？**
*   **解决“拉不动”问题**：如果你把不可拉元素的宽度设为 `STRONG` 的 `preferred`，它会跟“填满容器”的 `STRONG` 约束打架，导致拉伸分配不均。设为 `REQUIRED` 下限，它就稳如泰山。
*   **解决“缩过头”问题**：通过 `REQUIRED` 的 `min_width` 保护，确保无论怎么缩，零件都不会变成 0 宽导致机床报错。

### 裁决三：一个被所有人遗漏但极其重要的点

**回复7 指出的"行高计算"问题，必须纳入 Phase 2。**

当前代码中：
```rust
row_height = row_height.max(elem.footprint_height());
```

当 `VAlign::Baseline` 时，这个计算是**错误的**。正确逻辑：

```
对于 Baseline 对齐的行：
  max_ascent = max(有 baseline 的元素: elem.baseline + elem.margin.top)
  max_descent = max(有 baseline 的元素: elem.height - elem.baseline + elem.margin.bottom)
  row_height = max_ascent + max_descent
  
  对于无 baseline 的元素（回退到底部）：
    baseline = height
    ascent = height + margin.top
    descent = margin.bottom
```

这样才能保证：**所有元素的基线在同一水平线上，且行高足够容纳所有元素的上伸部和下伸部。**
