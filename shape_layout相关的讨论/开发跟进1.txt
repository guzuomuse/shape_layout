#### 1. 激活 `VAlign`：行内垂直对齐（Top / Middle / Bottom）

* **现在的状况**：当前 `VAlign` 只是在 `rules.rs` 中预留，但在 `engine.rs` 中，所有元素的 `y` 坐标在解出来后，都被硬编码为了本行的起始 `y`（即顶部对齐）。
* **为什么它是神级功能**：
  在同一行内，我们经常会把一个大图片（高 30mm）和一个小文本（高 10mm）并排摆放。设计师非常需要小文本能够在本行内**垂直居中（Middle）**或**贴底对齐（Bottom）**。
* **实现复杂度**：**极低（不涉及 `kasuari`，纯算术）**。
  在 `engine.rs` 的最后收割结果时，我们根据 `config.valign`，动态对那些高度小于行高的元素进行 $Y$ 轴物理偏移即可：
  ```rust
  let mut final_y = y;
  if elem.height < row_height {
      match config.valign {
          VAlign::Top => {} // 默认，保持 y
          VAlign::Middle => {
              final_y = y + (row_height - elem.height) / 2.0;
          }
          VAlign::Bottom => {
              final_y = y + row_height - elem.height;
          }
      }
  }
  ```

---

#### 2. 激活 `Stretchable`：自适应横向拉伸（Figma 的 "Fill Container"）

* **现在的状况**：目前支持了 `shrinkable`（当空间不够时，可以往窄了缩）。但还没有优雅地支持 `stretchable`（当空间有富余时，元素自动往宽了拉伸，填满空白，即 Figma 的 **Fill** 属性）。
* **为什么它是神级功能**：

* **实现复杂度**：**低（利用 `kasuari` 现成的弱约束）**。

  ```rust
  if elem.constraints.stretchable {
      // 1. 硬性限制：它的宽度必须大于等于首选宽度 (REQUIRED)
      solver.add_constraints([(right_vars[i] - left_vars[i]) | GE(Strength::REQUIRED) | preferred_w]).unwrap();

      solver.add_constraints([right_vars[i] | EQ(Strength::WEAK) | interval_r]).unwrap();
  }
  ```

* **实现复杂度**：**极低**。

      pub width: f64,
      pub height: f64,

**Figma 级功能**：
*   如果用户设置了 **Hug Contents**，容器应该根据排版后的实际边界自动缩放（比如钥匙扣的外边框随名字长短自动变化）。

#### 4. “强制不换行”与“分组锁定” (Grouping & No-Wrap) —— 逻辑约束

## 四、最关键的三个缺失：文本 Auto-fit、基线对齐、元素 Margin

### 1. 文本 Auto-fit：可变数据的灵魂

**场景**：奖杯标签，名字从 "Tom"（3字符）到 "Christopher"（11字符）到 "亚里士多德·尼古拉耶维奇"（20字符）。

**当前行为**：宽度固定，文字溢出或截断。
**期望行为**：字号自动缩小，直到 `min_font_size`，若仍放不下则换行或报错。

```rust
// element.rs 需要扩展
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TextProperties {
    pub content: String,
    pub font_family: String,
    pub font_size: f64,           // 首选字号
    pub min_font_size: f64,       // 最小字号（Auto-fit 下限）
    pub max_font_size: f64,       // 最大字号（Auto-fit 上限）
    pub line_height: f64,         // 行高倍数（1.2, 1.5 等）
    pub auto_fit: AutoFitPolicy,
}

#[derive(Debug, Clone, Copy, Serialize, Deserialize)]
pub enum AutoFitPolicy {
    Fixed,           // 不缩放，溢出报错
    Shrink,          // 缩小字号，直到 min_font_size
    Wrap,            // 换行，不缩放字号
    ShrinkThenWrap,  // 先缩小，到 min_font_size 后换行
    Truncate,        // 截断并加省略号
}
```

**实现方式**：
- 用 `cosmic_text` 测量文本宽度
- 二分查找：给定宽度，找最大可用字号
- 或在 kasuari 中加变量：`font_size` 作为弱约束变量

### 2. 基线对齐：多行文本的"隐形网格"

**当前行为**：所有元素按 `y`（顶线）对齐。
**期望行为**：文本按基线对齐，图片按中线或底线对齐。

```
当前（错误）：
    ┌─────┐  ┌─────┐
    │ 文本 │  │ 图标 │   ← 顶线对齐，但文本"坐"在基线上，图标"悬"在空中
    └─────┘  └─────┘

期望（正确）：
    ┌─────┐
    │ 文本 │──────  ← 基线对齐：文本的基线 = 图标的视觉中线
    └─────┘  ┌─┐
             │图│
             └─┘
```

```rust
// rules.rs 扩展
#[derive(Debug, Clone, Copy, Serialize, Deserialize)]
pub enum VAlign {
    Top,        // 顶线对齐
    Baseline,   // 文本基线对齐（需要 cosmic_text 测量 ascent/descent）
    Center,     // 视觉中线对齐
    Bottom,     // 底线对齐
}
```

**实现方式**：
- `cosmic_text::Metrics` 提供 `ascent`, `descent`, `line_gap`
- 基线位置 = `y + ascent`（相对顶线）
- kasuari 约束：`elem_i.y + elem_i.ascent == elem_j.y + elem_j.ascent`

### 3. 元素 Margin：呼吸空间

**当前行为**：元素紧密排列，只有 `gap`（元素间间距）。
**期望行为**：每个元素可以有自己的 `margin`（外间距），类似 CSS。

```rust
// element.rs 扩展
#[derive(Debug, Clone, Serialize, Deserialize, Default)]
pub struct ElementMargin {
    pub top: f64,
    pub bottom: f64,
    pub left: f64,
    pub right: f64,
}

impl LayoutElement {
    pub margin: ElementMargin,  // 新增
    
    /// 有效尺寸（含 margin）
    pub fn effective_size_with_margin(&self) -> (f64, f64) {
        (
            self.width + self.margin.left + self.margin.right,
            self.height + self.margin.top + self.margin.bottom,
        )
    }
}
```

你这里用的是 首选宽度（preferred width） 来判断“是否放得下”。

问题在于：

• 但贪心阶段没有考虑 shrink 后的最小宽度

结果是：

✅ 这是安全的，不是错误  
❌ 但不是“最优紧凑”

这样将来你就不会怀疑是不是 bug。

二、还缺哪些“像 Figma / InDesign Frame 那样”的必要功能？

你问得特别对：  
“不是套料 nesting，而是排版 layout。”

下面我给你一个最小必要功能清单，按优先级排。

✅✅✅ 优先级 1（强烈建议）

1️⃣ Vertical Alignment（行内纵对齐）

你现在有 VAlign，但 Phase 1 没用。

工业排版里非常常见：

• 同一行里：

  • 文字（高 20）

  • 条码（高 40）

  • Logo（高 60）

用户期望：
• Top / Middle / Bottom 对齐

✅ 实现方式：
placed_y = row_y + match valign {
    Top => 0.0,
    Middle => (row_height - elem.height) / 2.0,
    Bottom => row_height - elem.height,
};


这是 Frame-based layout 的灵魂之一。

2️⃣ 元素 Margin（不是 padding）

现在只有全局 gap。

工业里经常需要：

• 元素 A 右边多留 5mm

• 条码离文字更远一点

✅ 建议加到 LayoutElement：
pub struct ElementConstraints {
    pub margin_left: f64,
    pub margin_right: f64,
    pub margin_top: f64,
    pub margin_bottom: f64,
}


kasuari 里：
left_i >= right_{i-1} + gap + margin_right_{i-1}


✅✅ 优先级 2（非常有价值）

3️⃣ 单行 / 单列强制换行（Paragraph Break）

现在只有“放不下才换行”。

但工业排版需要：
• 标题一行

• 空一行

• 正文

✅ 最简单实现：
pub struct LayoutElement {
    pub force_new_line: bool,
}


engine 里：
if elem.force_new_line {
    flush_current_row();
}


4️⃣ 元素 Anchor / Origin（不是旋转，是锚点）

Figma 的 Frame 支持：
• TopLeft / Center / BottomRight

你现在是：
• x / y = 左上角

将来加旋转 / 嵌套 Frame 时，会非常有用。

✅ 现在只定义，不实现逻辑：
pub enum Anchor {
    TopLeft,
    Center,
    BottomRight,
}


✅ 优先级 3（可延后）

功能 说明

Max lines / Truncation 超出截断（Phase 2）

Text auto-fit（字号缩小） 非 shrink width

RTL / Bidi 工业印刷暂时不需要

Z-index 异形排版暂不需要
