### 一、 核心 Bug 审计：【重度隐蔽】Fill 元素非末尾时的“悬空间距漏洞”

在 `engine.rs` 的 `solve_row_x` 中，你设计了两个关于间距（gap）和 Fill 元素的约束注入：

1. **基础间距不等式约束**（第 483-490 行）：
   ```rust
   // left_{i+1} >= right_i + gap (REQUIRED)
   solver.add_constraints([(left_vars[i + 1] - right_vars[i]) | GE(Strength::REQUIRED) | config.gap])
   ```
2. **Fill 元素紧跟前驱等式约束**（第 494-508 行）：
   ```rust
   // 只有当前元素是 Fill 时，才强制与前驱贴合
   if elem.constraints.size_strategy.can_stretch() {
       solver.add_constraints([(left_vars[i] - right_vars[i - 1]) | EQ(Strength::STRONG) | config.gap])
   }
   ```

#### 1. 致命的“自由度悬空”物理分析（Bug 场景）：
如果这一行里有多个元素，**但 `Fill` 元素不在末尾，而是在开头或中间**。
* **场景**：排入两个元素：`A`（`SizeStrategy::Fill`，可拉伸）和 `B`（`SizeStrategy::Fixed`，宽 20mm，不可拉伸）。
* **物理区间**：总宽度为 100mm，设定 `gap = 10mm`。
* **约束解析**：
  * 因为 `A`（Fill）在索引 0，它没有前驱（`i = 0`），所以**没有触发**等式约束；
  * 因为 `B`（Fixed）在索引 1，它不是 Fill，所以**也没有触发**等式约束；
  * **结果**：在整个约束方程组中，**没有任何等式约束去锁定 `B.left - A.right` 的精确间距**。方程组里只剩下一个 `B.left >= A.right + 10` 的不等式。

#### 2. 数学灾难（在 Simplex 矩阵中的表现）：
此时，由于 `right_last - left_0 == 100`（整行强行拉满），以及 `B.width == 20`（硬性限制）。
* 系统只限制了 `left_0 = 0` 且 `right_1 = 100`（从而 `left_1 = 80`）。
* 因为没有任何约束去拉紧 `B.left` 与 `A.right` 的间距（它只需要 $\ge 10$ 即可）。
* **结果**：求解器（`kasuari`）会由于自由度悬空，**随意地给 A 安排一个任意宽度**。
  * **表现 1**：它可能会让 `A`（Fill）的宽度缩水为 0，而让 `A` 与 `B` 之间的间距（Gap）膨胀为 80mm！
  * **表现 2**：在不同的硬件或浮点数微调下，这个间距会发生**无规则的随机漂移**。
  * **测试缺失**：在你的测试中，因为测试用例 `test_fill_absorbs_remainder` 正好把 Fill 放到了末尾（"small" 在左，"fill" 在右），所以碰巧避开了这个 Bug。

#### 3. 极简且完美的修复方案：
其实你不需要去区分“当前元素是不是 Fill”。**在标准的 Flexbox 或 Figma 布局中，任何相邻元素之间的默认间距，有且只有在分配了 `SpaceBetween/Justify` 弹性间距时才能膨胀。在普通对齐下，所有相邻元素的间距必须死锁在 `config.gap` 上！**

你只需要把这两个约束，合并并简化为：
```rust
// ── 修正后：所有相邻元素之间的间距，默认全部死锁在 config.gap 上（STRONG） ──
if n > 1 {
    for i in 0..(n - 1) {
        // 1. 硬性底线：间距绝不能小于 gap
        solver.add_constraints([
            (left_vars[i + 1] - right_vars[i]) | GE(Strength::REQUIRED) | config.gap
        ]).unwrap();

        // 2. 强力对准：间距偏好等于 gap
        // 这会强行锁死所有间距，把所有的“拉伸力”完美、干净地逼入到 Fill 元素的 width 中！
        // 同时也直接干掉了原来第 494-508 行那段复杂的、存在悬空漏洞的 "Fill紧跟前驱" 逻辑。
        solver.add_constraints([
            (left_vars[i + 1] - right_vars[i]) | EQ(Strength::STRONG) | config.gap
        ]).unwrap();
    }
}
```
*这样一改，无论 `Fill` 在左边、右边还是中间，哪怕有 3 个 Fill 夹在 5 个 Fixed 中间，`kasuari` 都能以最完美的物理表现，把剩余空间一尘不染地均分给所有可拉伸元素，Bug 彻底根除！*

  * **结果**：文字变长 ──> 自动折行 ──> 软件自动向下推开后面的元素，并重新做物理切片。

## 🚨 严重 Bug（必须修复）

### Bug 1: `SizeStrategy::Fill` 的 kasuari 约束实现错误

**位置**: `engine.rs::solve_row_x()` 第 340-345 行

```rust
SizeStrategy::Fill => {
    // 状态 4: 可缩可伸
    // 不添加宽度相等约束（WEAK 会导致 kasuari 通过推位置而非拉伸来满足）
    // 仅靠 min/max REQUIRED 边界 + STRONG row-fill 自然推动拉伸
    has_fill_in_row = true;
}
```

**问题**: 当元素是 `Fill` 时，**完全不添加任何宽度偏好约束**。这意味着 kasuari 没有任何理由去拉伸这个元素——它只会把元素压缩到 `min_width`（如果没有 `min_width` 就是 0），然后靠 `row-fill` 约束把剩余空间推到 `gap` 上。

**后果**: 
- 如果没有 `min_width`，`Fill` 元素宽度会变成 0
- 如果有 `min_width`，`Fill` 元素宽度会卡在 `min_width`，不会主动拉伸
- `row-fill` 约束会把空间分配到 gap 上（元素位置偏移），而不是元素宽度上

**正确做法**（四态真值表修正）:

```rust
SizeStrategy::Fill => {
    // 状态 4: 可缩可伸
    // == preferred (WEAK) — 偏好首选值，但可被拉伸/压缩覆盖
    solver.add_constraints(
        [(right_vars[i] - left_vars[i]) | EQ(Strength::WEAK) | footprint_w],
    )?;
    has_fill_in_row = true;
}
```

**为什么需要 WEAK**: 
- `WEAK` 比 `STRONG` 的 `row-fill` 和 `HAlign` 优先级低
- 当 `row-fill` 要求撑满时，`WEAK` 的宽度偏好会被覆盖，元素被拉伸
- 当 `max_width` 限制拉伸时，`REQUIRED` 的 `max_width` 会压过 `row-fill`，`WEAK` 宽度偏好生效，元素回到首选宽度

### Bug 2: `Fill` 元素的 "紧跟前驱约束" 逻辑错误

**位置**: `engine.rs::solve_row_x()` 第 395-410 行

```rust
// Fill 元素紧跟前驱约束
if has_fill_in_row && n > 1 {
    for i in 1..n {
        let elem = &elements[row_indices[i]];
        if elem.constraints.size_strategy.can_stretch() {
            solver.add_constraints(
                [(left_vars[i] - right_vars[i - 1]) | EQ(Strength::STRONG) | config.gap],
            )?;
        }
    }
}
```

**问题**: 这个约束**只检查当前元素是否是 Fill**，但**不检查前驱元素是否是 Fill**。如果前驱是 Fixed，这个约束会把 gap 锁死为精确值，导致 Fixed 元素无法移动来让 Fill 元素拉伸。

**更深层问题**: 这个约束和 `row-fill` 约束是**冗余且冲突的**。
- `gap` 约束已经是 `REQUIRED: left_{i+1} >= right_i + gap`
- 额外的 `EQ(Strength::STRONG)` 会把 gap 锁死为精确值
- 当 `row-fill` 要求撑满时，如果 Fixed 元素不能移动，Fill 元素只能通过拉伸来满足，但如果没有 `WEAK` 宽度偏好（Bug 1），kasuari 不知道往哪个方向拉伸

**建议**: **删除这段 "Fill 紧跟前驱约束" 代码**。它弊大于利。`gap` 的 `REQUIRED >=` 约束 + `row-fill` 的 `STRONG ==` 约束已经足够让 kasuari 自动分配空间。

### Bug 3: `Fill` + `HAlign::Center` 的 fallback 可能不工作

**位置**: `engine.rs::solve_row_x()` 第 420-425 行

```rust
if has_fill_in_row {
    let last = n - 1;
    solver.add_constraints(
        [(right_vars[last] - left_vars[0]) | EQ(Strength::STRONG) | interval_width],
    )?;
}
```

**问题**: `row-fill` 约束是 `STRONG`，而 `HAlign::Center` 约束也是 `STRONG`。当 `max_width` 阻止完全填满时（`REQUIRED` 压过 `STRONG`），kasuari 会在两个 `STRONG` 约束之间做**加权最小二乘平衡**，结果可能既不是完全填满也不是完美居中，而是某种"折中"状态。

**正确做法**: `row-fill` 约束应该用 `MEDIUM` 或 `STRONG` 但低于 `HAlign` 的 `STRONG`，或者反过来。需要明确优先级：

```
优先级链: REQUIRED > HAlign (STRONG) > row-fill (MEDIUM) > preferred width (WEAK)
```

这样当 `max_width` 阻止填满时：
1. `REQUIRED` 的 `max_width` 满足
2. `STRONG` 的 `HAlign::Center` 满足 → 元素组居中
3. `MEDIUM` 的 `row-fill` 无法满足 → 被放弃
4. `WEAK` 的 preferred width 满足 → 元素回到首选宽度

```489:505:d:\www\bevy_projects\pce\crates\shape_layout\src\engine.rs

                    .add_constraints(

    for i in 0..(n - 1) {
        // 1. 硬底线：gap 不能小于 config.gap（REQUIRED）
        solver
            .add_constraints(
                [(left_vars[i + 1] - right_vars[i]) | GE(Strength::REQUIRED) | config.gap],
            )
            .map_err(|e| format!("add_constraint gap>=min failed: {e:?}"))?;
        
        // 2. 偏好：gap 精确等于 config.gap（STRONG）
        //    确保所有"拉伸力"被推入 Fill 元素的宽度，而非膨胀间距
        solver
            .add_constraints(
                [(left_vars[i + 1] - right_vars[i]) | EQ(Strength::STRONG) | config.gap],
            )
            .map_err(|e| format!("add_constraint gap==exact failed: {e:?}"))?;
    }
}
```

然后删除现有的 `Fill 元素紧跟前驱约束` 代码块（第 489-505 行），因为它已被上面更全面的逻辑覆盖。

/// [Fill, Fixed] 排列：验证 Fill 在 Fixed 前面时仍然正确拉伸
#[test]
fn test_fill_before_fixed() {
    let container = square(0.0, 0.0, 200.0);
    let mut fill_elem = LayoutElement::new("fill", 20.0, 20.0);
    fill_elem.constraints.size_strategy = SizeStrategy::Fill;
    let fixed_elem = LayoutElement::new("fixed", 40.0, 20.0);
    
    let config = LayoutConfig::with_spacing(5.0, 5.0, 5.0);
    // 区间 = 190, Fixed 占 40, gap=5, Fill 应占 190-40-5=145
    
    let solution = layout_rows(&container, &[fill_elem, fixed_elem], &config);
    assert!(solution.is_fully_placed());
    let fill = &solution.placed[0];
    let fixed = &solution.placed[1];
    
    // Fixed 保持在 40
    assert!((fixed.width - 40.0).abs() < 1.0);
    // Fill 吸收了剩余空间
    assert!(fill.width > 40.0, "Fill should absorb space, got {}", fill.width);
    // gap = fill.right 到 fixed.left
    let gap = fixed.x - (fill.x + fill.width);
    assert!((gap - 5.0).abs() < 1.0, "gap should be ~5, got {}", gap);
}
