### 一、 核心 Bug 审计与修复建议

#### 1. 【重特大 Bug】绝对原点偏移错误（Origin Offset Bug）
* **文件**：`engine.rs`
* **代码行**：
  ```rust
  let mut y = config.padding_top;
  ```
* **问题分析**：
  这里假设了容器的包围盒（AABB）的顶部 $Y$ 轴坐标一定是从 `0.0` 开始的。在你的测试用例中，因为使用的是 `square(0.0, 0.0, 100.0)`，所以没有暴露问题。
  **但是，如果用户的异形容器在画布中间，例如其 AABB 为 `y0 = 150.0, y1 = 250.0`**：
  由于 `y` 依然从 `config.padding_top`（比如 10.0）开始算起，**切片矩形完全落在了异形容器的物理范围之外**！扫描线将返回空区间，整个排版系统会直接报废（判定空间不足）。
* **修复方案**：
  让 Y 轴的起始值相对于容器的实际物理顶部（`rg.extents.y0`）进行偏移：
  ```rust
  // 修正前
  let mut y = config.padding_top;
  // 修正后
  let mut y = rg.extents.y0 + config.padding_top;
  ```

---

#### 2. 【边缘 Case 缺陷】实际行高重构后的“物理越界”风险
* **文件**：`engine.rs`
* **代码行**：
  ```rust
  // 用实际行高重新查询区间（行内可能有更高元素改变了有效高度）
  let refined_row = rg.get_intervals_at(y, row_height, config.min_width);
  if !refined_row.is_empty() {
      if let Some(widest_refined) = refined_row.intervals.iter().max_by(...) {
          interval_l = widest_refined.0 + config.padding_left;
          interval_r = widest_refined.1 - config.padding_right;
      }
  }
  ```
* **问题分析**：
  这是一个极具隐蔽性的几何越界漏洞。
  在 narrowing（向下收缩的沙漏形、倒三角形）容器中：
  * 初始预估行高为 $H_{start} = 10$，计算出可用宽度为 `50`。
  * 贪心算法塞入了一个高度为 $25$ 的高元素，行高重构为 $H_{row} = 25$。
  * 重新调用 `refined_row` 查询，因为容器向下收缩，在 $H=25$ 的切片厚度内，**由于瓶颈效应，最宽区间可能已经缩水到了 `20`，甚至是 `row_range.is_empty()`（完全放不下）**。
  * 按照当前代码，如果 `refined_row.is_empty()`，它会**直接保持原有的、较宽的 `interval_l/r` 不变**。
  * **后果**：`kasuari` 依然使用较宽的边界求解，导致那个高元素**直接在物理上刺穿了收缩的容器边界，产生物理越界**。
* **修复方案**：
  如果 `refined_row` 返回的可用区间无法容纳已经打包的元素，我们需要将其安全降级（判定为当前 Y 轴放不下，推进 Y 重新寻找空间，或者抛出警告）：
  ```rust
  let refined_row = rg.get_intervals_at(y, row_height, config.min_width);
  if !refined_row.is_empty() {
      if let Some(widest_refined) = refined_row.intervals.iter().max_by(...) {
          let refined_w = widest_refined.1 - widest_refined.0 - config.padding_left - config.padding_right;
          // 确保重构后的宽度依然放得下已经打包的元素
          if refined_w >= used_width {
              interval_l = widest_refined.0 + config.padding_left;
              interval_r = widest_refined.1 - config.padding_right;
          } else {
              // 宽度不够了，放弃本行，推进 Y 重新寻找
              y += 0.5;
              continue;
          }
      }
  } else {
      // 变高后完全放不下了，推进 Y
      y += 0.5;
      continue;
  }
  ```

#### 3. 【约束漏洞】不可缩元素（Non-shrinkable）的宽度硬性保护
* **文件**：`engine.rs`
* **代码行**：
  ```rust
  // right_i - left_i == preferred_width (STRONG)
  solver
      .add_constraints(
          [(right_vars[i] - left_vars[i]) | EQ(Strength::STRONG) | preferred_w],
      )
  ```
* **问题分析**：
  虽然 `pack_row_elements` 在前置分行时保证了不超重，但在一些极端边界情况下（如微小的浮点数公差或对齐推挤），如果空间稍微拥挤：
  * 因为你给宽度设定的约束是 `Strength::STRONG`（非强约束 `REQUIRED`），而边界安全约束 `left >= L` 和 `right <= R` 是 `REQUIRED`。
  * `kasuari` 为了满足 `REQUIRED`，**会强行缩短哪怕是 `shrinkable = false` 的元素的宽度**来苟活。这在物理制造中是不允许的。
* **修复方案**：
  在注入宽度约束时，根据元素的 `shrinkable` 属性，动态给 `kasuari` 施加严格的 `REQUIRED` 锁：
  ```rust
  let width_strength = if elem.constraints.shrinkable {
      Strength::STRONG // 允许弹性伸缩
  } else {
      Strength::REQUIRED // 绝对不可缩，必须死锁首选宽度
  };

  solver
      .add_constraints(
          [(right_vars[i] - left_vars[i]) | EQ(width_strength) | preferred_w],
      )
  ```

---

#### 4. 【输入校验建议】自相矛盾约束的预防御
* **文件**：`element.rs`
* **建议**：
  如果用户/AI 传入的 `min_width` 大于 `max_width`，`effective_width` 计算会锁死在 `min_width`，但会造成后期 `kasuari` 求解时发生致命冲突（`ConstraintConflict`）。
  建议在 `LayoutElement` 的构造器中加入简单的防御：
  ```rust
  if let (Some(min), Some(max)) = (self.constraints.min_width, self.constraints.max_width) {
      assert!(min <= max, "min_width cannot be greater than max_width");
  }
  ```

### 一、 源码逻辑审查（Code Review）

#### 1. 致命步长问题：`y += 0.5`
在 `engine.rs` 的 `layout_rows` 循环中，当遇到空行或放不下时，你使用了 `y += 0.5`。
*   **潜在风险**：这个 `0.5` 是一个“魔数”。
    *   如果用户单位是**毫米**，0.5mm 步进探测尚可。
    *   如果用户单位是**厘米**，0.5cm 太大，会跳过很多精细空间。
    *   如果用户是做**微雕**，0.5mm 又太大了。
*   **建议**：在 `LayoutConfig` 中增加一个 `step_size`（步进精度），默认可以设为 `0.1` 或者 `row_height * 0.1`。让探测精度可控。

#### 2. 贪心打包与 kasuari 的“权限冲突”
在 `pack_row_elements`（贪心）和 `solve_row_x`（大脑）之间存在一个微小的逻辑裂缝：
*   **现状**：`pack_row_elements` 决定谁进这行，它是基于“首选宽度”来算的。
*   **风险**：如果一个元素设置了 `shrinkable: true` 且 `min_width` 很小。`pack_row_elements` 可能会因为它能缩得极小，而把过多的元素塞进一行。
*   **结果**：导致 `solve_row_x` 虽然能解出来，但结果可能导致所有元素都缩成了最小宽度，变得极其难看。
*   **建议**：在 `solve_row_x` 中，给 `preferred_width` 施加的 `STRONG` 强度是正确的，但建议引入 **“弹性权重”**。比如让 `LayoutElement` 拥有一个 `shrink_priority`，让 AI 可以指定谁先缩、谁后缩。

#### 3. “最宽区间”的局限性
*   **现状**：`layout_rows` 目前在每一行只取一个 `widest` 最宽区间。
*   **风险**：如果容器是一个“U”字形，或者中间有一个巨大的孔洞。同一行 Y 上其实有两个并排的可加工区域。
*   **结果**：你目前的代码会忽略掉其中一个区域，造成严重的**空间浪费**。
*   **建议 (Phase 2)**：不要只取最宽，而是要遍历 `row_range.intervals`。如果一个区间放不下了，尝试把剩下的元素放到同行的下一个区间里。这才是真正的“异形排版”。

#### 4. “不可缩”元素的保护
*   **现状**：如果元素 `shrinkable` 为 `false`，在 `solve_row_x` 里它依然只有 `EQ(Strength::STRONG)`。
*   **风险**：如果 `kasuari` 面对一个极窄的区间，为了保住 `REQUIRED` 的边界，它依然可能会把不该缩的元素给压扁（因为 `STRONG` 虽然强，但低于 `REQUIRED`）。
*   **建议**：如果 `shrinkable` 为 `false`，必须给它加一个 `(right - left) >= preferred_width | Strength::REQUIRED` 的硬性下限。

### 一、发现的主要 Bug / 问题（需优先修复）

#### 1. **pack_row_elements 中 used_width 计算有逻辑漏洞**（最严重）

```rust
let total_needed = used_width + gap_needed + preferred_w;
if total_needed <= available_width ... {
    row_indices.push(idx);
    used_width = total_needed;   // ← 这里有问题
```

**问题**：`used_width` 被累加成了 **包含 gap 的值**，但下一次循环 `gap_needed` 又会重新加上 gap，导致**间距被重复计算**。

**修复建议**（推荐）：

```rust
let mut used_width = 0.0;   // 只记录元素实际占用宽度（不含 gap）

while idx < elements.len() {
    let elem = &elements[idx];
    let preferred_w = elem.effective_width();
    let gap_needed = if row_indices.is_empty() { 0.0 } else { config.gap };

    let total_needed = used_width + gap_needed + preferred_w;

    if total_needed > available_width + 1e-9 {
        // 尝试 shrink ...
    }

    // 成功放入
    row_indices.push(idx);
    used_width += preferred_w;          // 只加元素宽度！
    row_height = row_height.max(elem.height);
    idx += 1;
}
```

#### 2. **solve_row_x 中 Center 对齐逻辑有缺陷**

当前 Center 对齐只约束了 `left_0 + right_last`，但**中间元素的间距和对齐没有强制**，容易出现元素堆在一起或间距不均。

**改进建议**：
- 对于 Center，最好**先用 Left 对齐求解**，再整体平移到中心；或者增加所有元素相对于行中心的对称约束（更复杂）。
- 或者增加一个 `distribute`（均匀分布）模式。

#### 3. **行高动态更新后可能导致区间不准**

你现在在 pack_row_elements 后重新查询 `refined_row`，这很好。但如果新行高**显著增大**，可能导致原来认为可用的区间实际上已经被更高位置的孔洞遮挡。

**建议**：在 `refined_row.is_empty()` 时，应该**回退**或**降低行高尝试**，而不是直接使用可能无效的区间。

#### 4. **较小问题**
- `find_bottleneck` 中翻转 contour 后没有更新 `signed_area` 判断，可能有重复翻转风险。
- `layout_rows` 中 `y += 0.5` 的 magic number 建议改成 `config.min_line_advance` 或常量。

## 二、发现的潜在问题（建议修复后再进入 Phase 2）

### Bug 1：`pack_row_elements` 中的 `used_width` 计算有误

```rust
// 当前代码（有问题）
let total_needed = used_width + gap_needed + preferred_w;
// ...
used_width = total_needed;
```

**问题**：`used_width` 被累加为“左侧起点 + 所有元素宽度 + 所有间距”。但下一次迭代时，`gap_needed` 又加了一次，导致**间距被重复计算**。

**正确逻辑**：`used_width` 应该只累加**元素宽度**，间隙单独计算。

**建议修复**：

```rust
let mut used_width = 0.0; // 只累计元素宽度
// 在循环中：
let gap_needed = if row_indices.is_empty() { 0.0 } else { config.gap };
let total_needed = used_width + gap_needed + preferred_w;
// ...
used_width += preferred_w; // 只加元素宽度，不加 gap
```

### Bug 2：`solve_row_x` 中 Center 对齐的约束可能无解

```rust
HAlign::Center => {
    let target = interval_l + interval_r;
    solver.add_constraints(
        [(left_vars[0] + right_vars[last]) | EQ(Strength::STRONG) | target]
    )?;
}
```

**问题**：当元素总宽度 + 间距 > 可用宽度时，这个约束会与宽度约束和间距约束冲突，导致求解失败。

**建议**：将 Center 对齐的强度从 `STRONG` 降为 `MEDIUM` 或 `WEAK`，或者增加一个“如果无解则自动左对齐”的 fallback 逻辑。

### Bug 3：行高动态更新后，Y 推进可能跳过可用空间

```rust
let row_height = elements[idx].height; // 初始预估
// ... 放入元素，可能更新 row_height
let refined_row = rg.get_intervals_at(y, row_height, ...); // 用实际行高重查
// ...
y += row_height + config.line_spacing;
```

**问题**：如果 `row_height` 在放入元素后变大（因为某个元素更高），第一次查询用的行高是旧的，可能导致该行某个区间被低估，错过了一些可用空间。

**建议**：在放入元素后，用实际行高重新查询，并在新查询结果中重新放置当前行。这不是致命 bug，但在密集排版时可能导致次优结果。可以在 Phase 2 中优化。

### 设计建议：`HAlign::Center` 在单元素场景下的语义

当前代码对单元素 Center 对齐有效，但双元素 Center 对齐的含义是“整行居中”。这个语义是清晰的，但在文档中可以补充说明。

## 一、关键Bug #1：`pack_row_elements` 的缩放行高计算错误

```rust
// engine.rs: pack_row_elements 第 87-97 行
} else if elem.constraints.shrinkable {
    let min_w = elem.constraints.min_width.unwrap_or(0.0).max(0.0);
    let total_min = used_width + gap_needed + min_w;
    
    if total_min <= available_width + 1e-9 {
        row_indices.push(idx);
        used_width = total_min;  // ❌ BUG：记录的是 min_w，不是实际宽度
        row_height = row_height.max(elem.height);
        idx += 1;
    }
}
```

**问题**：`used_width` 记录的是 `min_w`，但 kasuari 求解时可能给的是 `preferred_w`（如果放得下）。这导致：
- 行内总宽度计算错误
- 后续元素可能"以为"还有空间，实际 kasuari 求解时溢出

**修复**：
```rust
if total_min <= available_width + 1e-9 {
    row_indices.push(idx);
    // 记录实际需要的宽度：preferred 或 min（取决于能否放下 preferred）
    let actual_w = if total_needed <= available_width + 1e-9 {
        preferred_w  // 能放下 preferred，用 preferred
    } else {
        min_w  // 只能放下 min，用 min
    };
    used_width = used_width + gap_needed + actual_w;
    row_height = row_height.max(elem.height);
    idx += 1;
}
```

## 二、关键Bug #2：`solve_row_x` 中 `right_vars` 是冗余变量

```rust

let mut left_vars: Vec<Variable> = Vec::with_capacity(n);
let mut width_vars: Vec<Variable> = Vec::with_capacity(n);  // 如果需要 stretch

for _ in 0..n {
    left_vars.push(Variable::new());
    width_vars.push(Variable::new());  // 宽度也是变量（支持 stretch/shrink）
}
```

## 三、架构隐患 #1：`row_height` 的两次查询不一致

```rust
// engine.rs: layout_rows 第 45-75 行
let mut row_height = elements[idx].height;  // 初始：第一个元素高度

// ... 贪心打包后 ...
let (row_indices, new_idx, final_row_height) = pack_row_elements(...);
row_height = final_row_height;  // 更新为行内最大高度

// 用实际行高重新查询区间
let refined_row = rg.get_intervals_at(y, row_height, config.min_width);
```

**问题**：
1. `pack_row_elements` 的 `final_row_height` 是**打包前**计算的（基于元素原始高度）
2. 但 kasuari 求解后，元素宽度可能变化（shrink/stretch），**不影响高度**
3. 真正的问题：**行高应该由"该行实际放置的元素"决定，但 `pack_row_elements` 已经决定了哪些元素进这一行**

**更深层问题**：如果 `refined_row` 的区间比第一次查询的**窄**（因为行高增加，遇到孔洞），可能导致 kasuari 求解失败。

**建议**：先查询区间，再打包，再求解。或者：**行高预估 → 查询区间 → 打包（用预估行高）→ 求解 → 如果失败，用实际行高重新查询**。

一、必须指出的几个“真实隐患”（请务必修）

这些不是“风格问题”，而是将来一定会炸的地方。

⚠️ 1. pack_row_elements 里的高度更新逻辑是“伪动态”

你现在是这样：
let mut row_height = elements[idx].height;
// ...
row_height = row_height.max(elem.height);


问题在于：

• 行高只根据“已塞入元素”的最大值更新

• 但 下一行元素的高度可能更大

• 你没有在塞入前预判“下一元素会不会把行高撑爆”

后果

会出现“元素在 Y 方向溢出异形区间”的风险。

建议修正（最小侵入）

在 pack_row_elements 里：
if row_height + elem.height > available_height {
    break;
}


或者在 get_intervals_at 之后，用最终行高再查一次区间（你其实已经做了 refined_row，很好，但要保证它生效）。

✅ 风险等级：中（必须修）

⚠️ 2. solve_row_x 中“中心对齐”约束在数学上是脆弱的

你现在用的是：
(left_vars[0] + right_vars[last]) | EQ | (interval_l + interval_r)


问题在于：

• 这是一个“等边距”约束

• 当元素总宽度 < 可用宽度时，这个约束是 OK 的

• 当元素总宽度 > 可用宽度时，kasuari 会无解或给出奇怪数值

建议

中心对齐应改为 “整体居中”，而不是“等边距”：
// 先算总宽度
let total_width = right_vars[last] - left_vars[0];

// 居中约束
(left_vars[0] + total_width / 2.0) | EQ | ((interval_l + interval_r) / 2.0)


✅ 风险等级：中（建议修）

⚠️ 3. ElementConstraints::stretchable 目前是“假 stretch”

你现在有字段，但没有在 kasuari 里使用它。
pub stretchable: bool,


而你的约束永远是：
(right - left) | EQ | preferred_width


这意味着：
• stretchable 元素 永远不会被拉伸

• 行内剩余空间不会被吃掉

建议（Phase 2 必须做）

if elem.constraints.stretchable {
    solver.add_constraint(
        (right_vars[i] - left_vars[i]) | EQ | available_width / n
    );
}


✅ 风险等级：低（Phase 2 修）

⚠️ 4. y += row_height + line_spacing 没有检查“是否还在异形内”

你已经做了：
if y >= max_y { break; }


但没有检查“下一行起始 y 是否仍在异形区间内”。

在极端异形（比如细脖子形状）里，可能出现：

上一行还在，下一行直接飘出轮廓

建议

在 layout_rows 主循环里加：
if y + row_height > max_y {
    // 整行丢弃，放入 unplaced
}


✅ 风险等级：中（建议修）
