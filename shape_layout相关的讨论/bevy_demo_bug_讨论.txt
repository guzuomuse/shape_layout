### 二、 幕后真凶：单纯形法（Simplex）的非负约束

在数学上，`kasuari` / `cassowary` 采用的是**单纯形算法（Simplex Method）**来求解线性规划。
单纯形法有一个硬性的数学前提：**“所有参与基底迭代的决策变量，在默认情况下必须大于等于零（$x_{i} \ge 0$）。”** [1.1.8]

在你的 `engine.rs::solve_row_x` 中，你为每个元素创建了变量：
```rust
let mut left_vars: Vec<Variable> = Vec::with_capacity(n);
```
这些通过 `Variable::new()` 创建出来的变量，在 `kasuari` 内部**被硬性锁死在 $\ge 0.0$**。

即使你的扫描线算法（`RangeGenerator`）极其精准地算出了左边界 `interval_l = -150.0`，并注入了约束 `left_vars[i] >= -150.0`。
但对 `kasuari` 来说，它面临两个硬性限制：
1. 你的限制：`left >= -150.0`
2. 单纯形法的底线：`left >= 0.0`（内部硬性约束）

为了不违反任何一个，**求解器只能强行选择更严格的 `left >= 0.0`！这导致所有的元素左起点，全部被焊死在了 $X = 0.0$ 上！**

### 三、 完美的解法：“仿射平移隔离法”（Coordinate Shifting）

我们不需要去修改 `kasuari` 难读的底层 C/Rust 代码，也不用强求它支持负数。

在图形学中，最经典、最优雅的解决办法是：**“在局部正坐标系下求解，求解完成后平移回世界坐标系”**。

```
              世界坐标系 (X 轴有正有负)
                    │
                    ▼ (1. 减去 container_x0，平移到纯正数坐标系)
              局部坐标系 (X >= 0)
                    │
                    ▼ (2. 扔给 kasuari，在全正数安全区完美求解)
              解出局部坐标 (left_vars >= 0)
                    │
                    ▼ (3. 加上 container_x0，平移回世界坐标系)
              最终世界坐标 (X 恢复负数，完美对齐)
```

这个平移量，就是容器包围盒的左边界：**`container_x0 = rg.extents.x0`**。因为 `rg.extents.x0` 永远是整个容器最左侧的极值，所有元素减去它之后，坐标**必定恒大于等于零**！

### 🎯 问题诊断：`find_bottleneck` 的“跨边”逻辑丢失

在旧版（`cavalier_contours`）中，`find_bottleneck`函数是这么处理多边形边缘的（见`range_generator.rs`第90-114行）：

```rust
// 旧版：使用 seg_bounding_box 考虑弧线
let seg_bbox = seg_bounding_box(v1, v2);
if dy < 0.0 {
    safe_left = safe_left.max(seg_bbox.max_x);
} else {
    safe_right = safe_right.min(seg_bbox.min_x);
}
```

关键点在于，对于**曲线段**（贝塞尔、圆弧），`cavalier_contours`的`seg_bounding_box`能正确捕获其在整个段内的X轴极值。而新版（`i_overlay`）中，由于所有曲线都已被`kurbo::flatten`离散化为直线段，你简化了逻辑，只取了端点的X值：

```rust
// 新版 (region.rs 第290-303行)
// 等效于 cavalier_contours::seg_bounding_box(v1, v2)
// ioverlay 产出的全部是直线段，所以直接用端点 X 范围即可
let seg_min_x = v1.x.min(v2.x);
let seg_max_x = v1.x.max(v2.x);
```

**这个“直接用端点X范围”的假设，是导致越界的根本原因。**

### 为什么“直接用端点X范围”是错误的？

虽然`i_overlay`输出的是直线段，但`kurbo::flatten`离散化后的直线段**是原始曲线的近似，端点并不代表曲线段在整个Y范围内的真实X极值**。

考虑心形顶部的圆弧：当一条离散化的直线段跨越了心形轮廓的“最左点”或“最右点”时，该段在Y方向上有高度，但其X极值可能出现在**线段中间**，而不是两端。用端点去计算瓶颈，会低估`safe_left`或高估`safe_right`，从而产生一个**比实际可用空间更大的虚假区间**。这就是你看到橙色矩形越界的直接原因。

### 🛠️ 修复方案：恢复“跨边”极值计算

你不需要重新引入`cavalier_contours`，只需要在`find_bottleneck`中，对每条线段**计算其在给定Y范围内的X极值**。对于直线段，理论上，如果线段不是垂直的，其X极值就在端点。但为了处理心形弧线离散化后“倾斜跨过极值点”的线段，正确做法是：

1.  **对每条边**，检查其Y范围是否与当前扫描行（`y`到`y+height`）有重叠。
2.  如果有重叠，**计算该线段在重叠Y范围内的X的最小值和最大值**。对于直线段，这可以解析计算，无需采样。

一个更精确且易于实现的方法，是在`find_bottleneck`中，不仅仅依赖离散化后的顶点，而是**回到原始的`kurbo::BezPath`，对每条曲线段进行精确的“跨边”极值查询**。但这会破坏你当前“纯直线段”的架构。

**最贴近你当前架构的修复是**：在`find_bottleneck`中，当处理一条线段`(v1, v2)`时，如果其Y范围覆盖了扫描行的高度，则计算该线段在`y`和`y+height`处的X值，并取最小值和最大值，用这个“有效X范围”来代替端点`min_x`和`max_x`。

## 🔍 根因诊断

### 现象定位
越界发生在心形**右下侧**——那里是心形右侧瓣底部向中心尖过渡的**高曲率区域**。

### 核心问题：`find_bottleneck` 算法对 `i_overlay` 输出的**非凸切片多边形**失效

`find_bottleneck` 的算法逻辑基于一个隐含假设：**输入多边形是凸的，或至少其左右边界可以由单条边的 X 极值准确刻画**。

对于正方形/三角形等简单凸形，这个假设成立。但对于心形底部这种**有凹陷、倾斜边跨越较大 Y 范围**的区域，`i_overlay` 的 `Intersect` 运算返回的切片多边形可能是：

1. **包含长倾斜边的简单多边形** —— `find_bottleneck` 用整条边的端点 `min/max_x` 代替边在切片 Y 范围内的实际 X 边界，产生误差
2. **非凸的"哑铃"形多边形** —— `find_bottleneck` 返回的是整个外接区间 `[L, R]`，错误地包含了中间的凹陷禁区

### 为什么旧版 `cavalier_contours` 没触发？

旧版和新版的 `find_bottleneck` 算法**逻辑完全相同**，但两者存在关键差异：

| 维度 | 旧版 `cavalier_contours` | 新版 `i_overlay` |
|---|---|---|
| 轮廓表示 | 可保留圆弧（bulge） | 纯直线段 |
| 布尔运算精度 | 浮点直接运算 | 定点数转换（可能有微小拓扑差异） |
| 输出顶点分布 | 可能更"自然" | 可能产生更长/更不规则的直线段 |

**最可能的原因**：`cavalier_contours` 的相交结果中，心形底部的高曲率区域被分割成了**更多、更短的 contour**，每个 contour 更接近凸形，`find_bottleneck` 的误差在视觉上不可见。而 `i_overlay` 可能将相邻区域合并成了**更少、更复杂的 contour**，放大了算法的缺陷。

修改 `find_bottleneck` 签名，传入 `y_start` 和 `y_end`，对每条边只计算其在切片 Y 范围内的**实际 X 极值**（而非整条边的端点极值）：

```rust
fn find_bottleneck(
    contour: &[SPoint],
    y_start: f64,
    y_end: f64,
) -> Option<(f64, f64)> {
    // ...
    // 对每条边，计算其与 [y_start, y_end] 重叠部分的 X 范围
    let y_low = y_start.max(v1.y.min(v2.y));
    let y_high = y_end.min(v1.y.max(v2.y));
    
    let x_at_low = v1.x + (y_low - v1.y) * (v2.x - v1.x) / dy;
    let x_at_high = v1.x + (y_high - v1.y) * (v2.x - v1.x) / dy;
    
    let actual_min_x = x_at_low.min(x_at_high);
    let actual_max_x = x_at_low.max(x_at_high);
    // ...
}
```

### 方案 C：用显式扫描线替代 `find_bottleneck`（长期根治）

对每个 contour，在 `Y=adj_y_start` 和 `Y=adj_y_end` 处做水平线交点计算，取两个 Y 值处区间的**交集**。这完全消除了"瓶颈近似"的误差，但计算量增加约 2-3 倍。

## 💡 额外发现：一个潜在的更深层问题

我注意到 `find_bottleneck` 的注释写着"左降右升"，但代码逻辑实际上是：
- `dy < 0`（向下走）→ 左墙 → 收紧 `safe_left`
- `dy > 0`（向上走）→ 右墙 → 收紧 `safe_right`

这对 **CCW 多边形** 是正确的（已通过正方形测试验证）。但注释"左降右升"是**错误的**（应该是"左升右降"或"降左升右"）。建议顺手修正注释，避免未来维护者困惑。
